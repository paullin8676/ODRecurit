# 表格结构更新 - 产品需求文档

## Overview
- **Summary**: 更新机考管理、韧测管理、面试管理模块的表格结构，使其符合业务需求：前三列保持与候选录入一致，面试管理改为弹框编辑，删除韧测总分字段，列出面试管理每个阶段的主要字段。
- **Purpose**: 优化用户体验，确保表格结构的一致性和合理性，提高数据管理效率。
- **Target Users**: 招聘管理系统的管理员和操作人员。

## Goals
- 机考管理、韧测管理、面试管理模块的前三列保持与候选录入一致（姓名、性别、手机号）。
- 面试管理模块的编辑方式改为弹框编辑，因为涉及字段和阶段太多，行内编辑会导致滚动条过长。
- 韧性测试表和前端列表删除韧测总分字段。
- 面试管理前端表格列出每个阶段的主要字段。

## Non-Goals (Out of Scope)
- 不修改现有数据结构和数据库表设计。
- 不改变系统的整体架构和技术栈。
- 不修改其他模块的功能和实现。

## Background & Context
- 系统使用 Vue.js + Element Plus 构建前端，Express.js + Sequelize 构建后端。
- 目前候选录入模块的前三列是：姓名、性别、手机号。
- 面试管理模块涉及多个阶段和字段，行内编辑会导致表格过长，影响用户体验。
- 韧性测试模块的总分字段可能不需要在表格中显示。
- 面试管理模块需要在表格中显示每个阶段的主要字段，以便快速查看。

## Functional Requirements
- **FR-1**: 机考管理模块的前三列保持与候选录入一致（姓名、性别、手机号）。
- **FR-2**: 韧测管理模块的前三列保持与候选录入一致（姓名、性别、手机号）。
- **FR-3**: 面试管理模块的前三列保持与候选录入一致（姓名、性别、手机号）。
- **FR-4**: 面试管理模块的编辑方式改为弹框编辑。
- **FR-5**: 韧性测试表和前端列表删除韧测总分字段。
- **FR-6**: 面试管理前端表格列出每个阶段的主要字段。

## Non-Functional Requirements
- **NFR-1**: 表格结构应保持一致性和美观性，符合 Element Plus 的设计规范。
- **NFR-2**: 编辑操作应响应迅速，避免页面刷新。
- **NFR-3**: 系统应保持稳定，避免因表格结构变更导致的错误和异常。

## Constraints
- **Technical**: 基于现有的 Vue.js + Element Plus 技术栈，不引入新的依赖。
- **Business**: 保持与现有业务流程的一致性，确保功能变更不影响正常的招聘管理流程。
- **Dependencies**: 依赖现有的 API 接口和数据结构，不修改后端实现。

## Assumptions
- 系统已经实现了基本的 CRUD 操作和数据验证。
- 各模块的 API 接口已经完善，能够支持表格结构变更的需求。
- 用户已经熟悉基本的表格操作和编辑流程。

## Acceptance Criteria

### AC-1: 机考管理表格结构
- **Given**: 用户打开机考管理模块
- **When**: 系统加载机考管理表格
- **Then**: 表格的前三列应与候选录入一致（姓名、性别、手机号）
- **Verification**: `human-judgment`

### AC-2: 韧测管理表格结构
- **Given**: 用户打开韧测管理模块
- **When**: 系统加载韧测管理表格
- **Then**: 表格的前三列应与候选录入一致（姓名、性别、手机号），且删除了韧测总分字段
- **Verification**: `human-judgment`

### AC-3: 面试管理表格结构
- **Given**: 用户打开面试管理模块
- **When**: 系统加载面试管理表格
- **Then**: 表格的前三列应与候选录入一致（姓名、性别、手机号），且列出了每个阶段的主要字段
- **Verification**: `human-judgment`

### AC-4: 面试管理编辑方式
- **Given**: 用户点击面试管理模块的编辑按钮
- **When**: 系统响应编辑操作
- **Then**: 应弹出编辑弹框，而不是行内编辑
- **Verification**: `human-judgment`
